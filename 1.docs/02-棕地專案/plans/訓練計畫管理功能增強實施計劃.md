# 訓練計畫管理功能增強實施計劃

**建立日期**: 2026-01  
**狀態**: 已完成

## 目標

增強訓練計畫管理模組，包含：個人授課對象設定、列表頁籤與篩選、封存功能、分類顯示與排序、操作選單與確認流程等，以提升管理與查詢效率。

## 需求來源

依據《教育訓練線上考卷系統 - 需求分析》第 8 點「訓練計畫管理」之項目。

## 一、資料層與後端

### 1.1 資料庫變更

**遷移腳本**：`backend/migrations/add_training_plan_enhancements.py`

- **plan_target_users**（多對多關聯表）
  - `plan_id`（FK → training_plans.id）
  - `emp_id`（FK → users.emp_id）
  - 複合主鍵：`(plan_id, emp_id)`

- **training_plans 新增欄位**
  - `is_archived`：INTEGER，預設 0，表示是否已封存

### 1.2 後端模型與結構

**檔案**：`backend/app/models.py`

- 新增關聯表 `plan_target_users`
- `TrainingPlan` 新增：
  - `target_users`：與 User 的 many-to-many
  - `is_archived`：Boolean，預設 False

**檔案**：`backend/app/schemas.py`

- `TrainingPlanCreate`：新增 `target_user_ids: List[str] = []`
- `TrainingPlan`：新增 `target_users`、`is_archived`、`sub_category`（用於避免「未知分類」）

### 1.3 API 行為

**檔案**：`backend/app/routers/training.py`

- **GET /training/plans**
  - 查詢參數：`status`（active | expired | archived | all）、`year`、`dept_id`、`category_id`
  - 預設僅過濾 `is_archived == False`，不預設過濾過期狀態（向後相容）
  - 使用 `joinedload(TrainingPlan.sub_category)` 載入分類，確保列表分類名稱正確

- **POST/PUT 訓練計畫**
  - 依 `target_user_ids` 更新 `plan_target_users`，實作個人授課對象

- **POST /training/plans/{plan_id}/archive**  
  將該計畫 `is_archived` 設為 True

- **POST /training/plans/{plan_id}/unarchive**  
  將該計畫 `is_archived` 設為 False

**檔案**：`backend/app/routers/exam_center.py`

- `get_my_exams`、`start_exam` 等查詢/操作加入 `is_archived == False` 條件，避免已封存計畫出現在考試中心與開始考試流程

**檔案**：`backend/app/routers/report.py`

- 成績中心相關統計（overview、active_plans、expiring_plans、plan_popularity 等）查詢 TrainingPlan 時，一律過濾 `is_archived == False`

## 二、前端：訓練計畫管理

**檔案**：`frontend/src/components/admin/TrainingPlanManager.tsx`

### 2.1 狀態與資料

- **頁籤**：`activeTab`: 'active' | 'expired' | 'archived'
- **篩選**：`filterYear`、`filterDeptId`、`filterCategoryId`
- **列表**：`plans`（當前篩選結果）、`allPlans`（同 status 下全量，供年份等篩選選項使用）
- **排序**：`sortField`、`sortDirection`；可排序欄位含 `year`、`title`、`dept_name`、`category_name`，且支援「遞增 → 遞減 → 恢復原狀」循環
- **操作選單**：`openActionMenu`（當前開啟的計畫 id），用於「…」下拉選單

### 2.2 列表與篩選

- 三個頁籤：「正在進行中」、「已過期」、「已封存」
- 篩選區：年份、開課單位、分類；「清除篩選」按鈕
- 年份選項來自 `allPlans`，確保切換頁籤或篩選後，所有年份仍可選
- 表格欄位：項次、年份、計畫名稱、開課單位、開始/結束日期、分類名稱、報到統計、操作

### 2.3 排序

- 可排序欄位：年份、計畫名稱、開課單位、分類名稱
- 同一欄位點擊循環：遞增 → 遞減 → 恢復原狀（`sortField = null`）
- 分類名稱排序依 `plan.sub_category?.name`，空值以「未知分類」參與排序

### 2.4 操作選單（「…」）

- **封存計畫**：未封存時顯示；點擊後先 `window.confirm` 再呼叫 archive API
- **取消封存**：已封存時顯示；點擊後先 `window.confirm` 再呼叫 unarchive API
- **刪除計畫**：始終顯示；點擊後開啟既有的刪除確認流程
- 下拉選單與遮罩使用較高 z-index（如 z-[90]、z-[100]），並注意表格容器 `overflow`，避免選單被裁切或蓋住

### 2.5 編輯表單

- **受課對象**：維持既有「受課單位」多選
- **個人受課對象**：多選使用者，搜尋條件為員工編號、姓名、部門（placeholder：「搜尋員工編號、姓名或部門…」）
- 表單送出時一併傳送 `target_dept_ids`、`target_user_ids`

### 2.6 其他

- 報到統計、QR Code、應到人數等既有邏輯不變；重新載入計畫時，依當前 `activeTab` 與篩選條件更新 `plans` 與 `allPlans`。

## 六、測試要點

1. **個人授課對象**：新增/編輯計畫可多選個人，儲存後列表與 API 回傳一致；應到人數含部門與個人。
2. **頁籤與篩選**：三個頁籤切換正確；年份/單位/分類篩選生效；年份選項在已過期、已篩選下仍含所有年份。
3. **封存**：封存後自「正在進行中」「已過期」消失、出現在「已封存」；考卷工坊、考試中心、成績中心均不露出已封存計畫。
4. **取消封存**：在已封存頁籤可從「…」看到並執行「取消封存」，且會先出現確認框。
5. **封存前確認**：執行「封存計畫」前會出現確認對話框。
6. **排序**：年份、計畫名稱、開課單位、分類名稱可排序，且依序為遞增→遞減→恢復原狀。
7. **分類名稱**：列表無「未知分類」；若有 sub_category 則顯示名稱。
8. **題庫匯入**：關鍵字、題型、標籤篩選與題庫維護一致，便於在考試工坊中篩選題目。

## 七、注意事項

1. **向後相容**：GET /training/plans 未傳 status 時僅過濾已封存，不預設排除過期，避免既有呼叫方行為改變。
2. **應到人數**：同時有受課單位與個人時，後端需避免重複計算（同一人只算一次）。
3. **z-index 與 overflow**：操作選單需高於表格與外層容器，並避免父層 `overflow: hidden` 裁切下拉選單。
