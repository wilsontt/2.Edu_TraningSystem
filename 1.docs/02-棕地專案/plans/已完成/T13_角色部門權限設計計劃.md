# T13 角色部門權限設計計劃

依據討論與需求文件：
- `1.docs/02-棕地專案/規格與計畫/T13 增修功能實作PLAN_測試問題.md`

## 1. 目標

提供可維護、可擴充的「角色部門權限」機制，支援以下情境：
- Admin 固定擁有所有資料可視範圍，不可修改。
- 其他角色可設定為：
  - `all`：所有部門
  - `department`：登入者自己部門 + 角色額外勾選部門
  - `self`：僅本人

## 2. 規則定義（已確認）

### 2.1 部門可視範圍
- `all`：不限制部門（即全公司資料）
- `department`：
  - 可視部門 = `{登入者自己的 dept_id}` UNION `{角色設定勾選的 dept_ids}`
  - 自己部門永遠包含
- `self`：只看本人資料

### 2.2 使用者狀態
- 可視對象預設僅 `active` 使用者（在 `active_only=true` 的查詢場景）

### 2.3 Admin 特例
- Admin 角色在「角色部門權限」中不可修改，固定為 `all`
- API 回傳時，Admin 也應固定回傳 `scope_type=all`

## 3. 資料模型設計

## 3.1 既有表
- `role_department_scope_map`
  - `role_id`
  - `scope_type` (`all` / `department` / `self`)

### 3.2 新增表
- `role_department_scope_depts`
  - `role_id`
  - `dept_id`
  - 複合主鍵：`(role_id, dept_id)`
  - 用於儲存 `department` 模式下「額外可視部門」

## 4. API 設計

### 4.1 讀取角色部門權限
- `GET /api/admin/roles/{role_id}/department-scope`
- Response：
  - `role_id`
  - `scope_type`
  - `dept_ids: number[]`

### 4.2 更新角色部門權限
- `PUT /api/admin/roles/{role_id}/department-scope`
- Request：
  - `scope_type: "all" | "department" | "self"`
  - `dept_ids: number[]`

### 4.3 驗證與保護
- 驗證 `scope_type` 合法性
- 驗證 `dept_ids` 必須存在於 `departments`
- Admin 更新請求強制收斂為 `all`

## 5. 核心查詢邏輯

統一由 `access_scope` 計算可見 `emp_id`：
- `all` -> `None`（查詢不加限制）
- `department` -> 以 `dept_id in allowed_dept_ids` 查 `users.emp_id`
- `self` -> `[current_user.emp_id]`

其中 `allowed_dept_ids` 計算：
- 先放入 `current_user.dept_id`
- 再 union 角色設定表中的 `dept_ids`

## 6. 前端管理頁設計

檔案：`frontend/src/components/admin/RoleDepartmentScopeManager.tsx`

### 6.1 操作模式
- 左側：角色清單
- 右側：可視範圍設定（all / department / self）
- 與「權限管理」一致：
  - 有未儲存提示
  - 支援「儲存後切換角色」

### 6.2 Department 模式 UI
- 額外可視部門（可多選）
- 提示「自己所屬部門預設包含」
- 提供：
  - 全選
  - 不全選
- 清單採 2 欄顯示（RWD）
- 顯示「自己所屬部門（預設勾選、不可取消）」提示列

### 6.3 Admin UI 保護
- Admin 選項不可切換
- Admin 儲存按鈕禁用

## 7. 驗收案例

### 案例 A：業務角色
- 假設：
  - 登入者部門 `dept_id=2`
  - 角色額外勾選 `[3,4,5,6,7]`
- 期望：
  - 報到總覽 / 成績中心可見 `2,3,4,5,6,7`

### 案例 B：未設定額外部門
- `department` 模式但 `dept_ids=[]`
- 期望：
  - 只看登入者自己的部門

### 案例 C：Admin
- 期望：
  - 固定 `all`
  - 不可修改

## 8. 維運與擴充性

- 新增/刪除部門時，不需改程式碼：
  - `all` 模式天然涵蓋全部部門
  - `department` 模式由資料表關聯驅動
- 若部門被刪除，DB FK 關係可避免髒資料
- 權限規則集中在 `access_scope`，降低分散邏輯風險
