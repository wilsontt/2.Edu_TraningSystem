# AD 整合：系統管理者登入實作計劃 (PLAN)

**文件類型**：實作計劃（PLAN，僅設計，不含程式碼）
**建立日期**：2026-06-12
**狀態**：草案（方案乙已納入；技術設計見 [`20260612_AD整合_系統管理者登入_技術設計.md`](./20260612_AD整合_系統管理者登入_技術設計.md)）
**對應需求**：IT 管理者改以 AD 群組 `IT_Admin` 登入管理端；AD 斷線時曾成功 AD 登入之 IT 帳號可改以 **Email OTP 備援**；**保留單一 break-glass 本地 ADMIN** 作為最後手段，並套用 ISO 27001 密碼政策。

> 本計劃依使用者全域規範採 7 大結構：目的／範圍／權責／名詞解釋／作業內容／參考文件／使用表單。
>
> **與建議事項 PLAN 分工**：本文件負責**管理端 LDAPS 登入**；教材傳輸之 **NAS SMB 驗證**（不同協定、不同用途）見 [`20260618_教育訓練_建議事項_PLAN.md`](./20260618_教育訓練_建議事項_PLAN.md)。

## 1. 目的

1. 將「IT 系統管理者」之**身分驗證**改由企業 AD（Active Directory）負責，僅允許 AD 群組**`IT_Admin`** 之成員登入管理端。
2. **AD 斷線備援（方案乙）**：當 DC 不可達時，曾以 AD 成功登入且本地存有 **Email**（JIT 自 AD `mail` 同步）之 IT 管理帳號，可改以 **Email 驗證碼（OTP）** 登入管理端（路徑 D）。
3. **保留單一本地管理帳號（break-glass）**：作為 **最後手段**（無 JIT 紀錄、無 email、OTP 失敗或極端災難時），以**本地密碼**登入。此帳號**不可刪除、不可停用至無法登入**，並套用 ISO 27001 密碼政策（**僅適用 break-glass，不適用 AD 帳號**）。
4. **保留本地 RBAC**（角色、功能選單、部門可視範圍）作為**授權**依據；AD 僅負責「**是誰**」與「**能否進入管理端**」（AD 正常時）。
5. 一併清理現存安全債（JWT 金鑰硬編碼、驗證碼後門`0000`），完成後執行 **Security Review**。

> **重要前提（已確認）**：
> 1. 本系統本地使用者主鍵為 `emp_id`（純數字員工編號），而 **AD 登入帳號（sAMAccountName）≠ 員工編號**。
> 2. **AD `IT_Admin` 群組成員為純管理身分，不參與教育訓練、不需考試**。
> 3. 因此 AD 登入採 **JIT 自動建檔（免對照表）**：群組檢查通過即時以 `ad_username` 自動建立／更新一筆「管理帳號」（`emp_id = ad_username`），並標記為**非受訓對象**，排除於考試／受課／報到／成績統計之外。**不需維護 emp_id ↔ ad_username 對照表**；人員異動由 AD 群組單一來源控管（移出群組者下次登入即被拒）。詳見 §5.1、§5.3。

## 2. 範圍

### 2.1 涵蓋範圍（In Scope）

- 新增「管理端 AD 登入」流程（LDAPS bind + 群組檢查）；**AD 帳號輸入白名單**與 LDAP filter 跳脫（防 LDAP injection）。
- AD 驗證通過後之**JIT（Just-In-Time）自動建檔**：以 `ad_username` 為 `emp_id` 自動 upsert 管理帳號，同步 AD **`mail` → `users.email`**，掛管理角色並標記**非受訓對象**；免對照表。
- **AD 斷線 Email OTP 備援**（路徑 D）：SMTP 寄送驗證碼；僅 AD 不可達時啟用；**不存 AD 密碼**。
- **JIT 管理帳號之隔離**：排除於考試中心、受課對象挑選、報到與成績統計之外。
- **本地 break-glass 帳密登入**與**ISO 27001 密碼政策**（複雜度、長度 ≥ 12、90 天到期、失敗鎖定；**僅 break-glass**）。
- 認證設定**環境變數化**（AD、JWT、SMTP、OTP、密碼政策、後門開關）。
- 既有 `auth.py` 權限判斷與超級角色判定之相容處理；**關閉管理角色之免密登入路徑**。
- 完成後 **Security Review**（見技術設計 §11）。

### 2.2 不涵蓋範圍（Out of Scope）

- **一般員工登入維持現狀**（工號＋圖形驗證碼免密），本計劃不改動。
- AD 群組與本地角色之**細粒度多對多映射**（本期僅單一群組`IT_Admin` → 單一管理角色）。
- AD 密碼修改、AD 帳號生命週期管理（屬 AD／IT 既有流程）。
- **emp_id ↔ ad_username 對照表**：本期**不採用**（改 JIT 自動建檔）。
- **AD 管理帳號參與教育訓練／考試**：本期 IT_Admin 帳號**不受訓、不應考**。
- **多個本地管理帳號**：本期僅保留**單一** break-glass 本地 ADMIN，不另開多個 supervisor。
- **密碼歷史比對**（不重用前 N 次舊密碼）：本期**不實作**。
- SSO / OIDC（Entra ID、ADFS）：本期採**LDAPS**，OIDC 留待未來評估。

## 4. 名詞解釋

| 名詞                         | 說明                                                       |
| ---------------------------- | --------------------------------------------------------- |
| **AD**                 | Active Directory，企業目錄服務                                    |
| **LDAPS**              | LDAP over SSL/TLS（埠 636），加密的目錄查詢／驗證協定                |
| **bind**               | 以帳號密碼向 AD 進行繫結驗證，成功即代表帳密正確                       |
| **DN / Base DN**       | Distinguished Name；Base DN 為搜尋起點（如 `DC=yourco,DC=local`）  |
| **UPN**                | User Principal Name，如 `user@yourco.local`                      |
| **sAMAccountName**     | AD 的傳統登入帳號（如 `it01`）                                      |
| **memberOf**           | AD 使用者所屬群組清單屬性                                           |
| **JIT 佈建**           | 首次登入時即時於本地建立／更新使用者資料（本案 AD 管理帳號採此，免對照表） |
| **is_trainee**         | 是否受訓對象；AD 管理帳號為 false，排除於考試／受課／報到／統計            |
| **RBAC**               | 本系統角色基礎存取控制（角色＋功能＋部門範圍）                          |
| **break-glass**        | 緊急用本地管理帳號（`admin`）；**最後手段**；平時封存，密碼政策僅適用此帳號                     |
| **Email OTP 備援**   | 路徑 D；AD 斷線時以寄至 JIT 同步之 email 之驗證碼登入；`auth_src=email_fallback`              |
| **ISO 27001 密碼政策** | 本案指：**僅 break-glass** — 複雜度、長度 ≥ 12、90 天到期、失敗鎖定；AD 密碼由 DC 負責         |
| **emp_id**             | 本地使用者主鍵，純數字員工編號（與 AD 帳號不同）                        |
| **NAS SMB 驗證**       | 教材傳輸時以 SMB 連 NAS（可用 AD 網域帳號）；**非**本 PLAN 之 LDAPS 管理登入 |

## 5. 作業內容

### 5.1 整體流程

本系統管理端登入分**四條路徑**（A／B／D 為管理端；C 為員工），一般員工免密登入不變。

#### 路徑 A：IT 管理者 — AD 登入（日常）

```
管理端登入頁 → 選「AD 登入」→ POST /api/auth/login/admin
  1. 檢查 AD_ENABLED（false → 503「AD 整合未啟用」）
  2. 驗證 username 白名單（§5.1.1）；LDAPS bind（失敗 → 401）
  3. 讀取 memberOf，檢查 IT_Admin（否 → 403）
  4. JIT upsert 管理帳號（含 email；見 §5.1「JIT 自動建檔」）
  5. 簽發 JWT（auth_src=ad）
  ※ DC 連線失敗 → 503，前端提示可改走路徑 D
```

#### 路徑 D：IT 管理者 — Email OTP 備援（AD 斷線）

```
管理端登入頁 → AD 登入 503 後 →「Email 驗證登入」
  → POST /api/auth/login/admin/email/request { username }
  1. 檢查 AD_FALLBACK_EMAIL_ENABLED
  2. 確認 AD 目前不可達（否 → 400「請使用 AD 登入」）
  3. 查本地 JIT 管理帳號：有 email、active、曾 AD 登入且 last_login_at ≤ 30 天
  4. 產生 OTP、寄信至 users.email（頻率限制）
  → POST /api/auth/login/admin/email/verify { username, otp_code }
  5. 驗證 OTP → 簽發 JWT（auth_src=email_fallback）
```

> **限制**：斷線期間**無法**即時查 AD 群組；資格依「本地曾有 JIT 管理帳號 + 時間窗」控管。AD 恢復後 IT 應改回路徑 A。

#### 路徑 B：本地 ADMIN — 帳密登入（break-glass，最後手段；不受 AD_ENABLED 影響）

```
管理端登入頁 → 選「本地緊急登入」→ POST /api/auth/login/local
  1. 驗證 emp_id + password（bcrypt）
  2. 檢查 is_protected 且具管理角色
  3. 檢查 locked_until（鎖定中 → 423）
  4. 檢查密碼是否過期 90 天 → 強制改密碼流程（§5.4.2）【僅 break-glass】
  5. 簽發 JWT（auth_src=local）或「須改密」之有限 token
```

#### 路徑 C：一般員工 — 免密登入（不變）

```
POST /api/auth/login（工號＋驗證碼）
```

#### 管理角色免密登入關閉

- 具**管理端功能**之角色（如`系統管理`、`Admin`），**不得**再經路徑 C 登入。
- 路徑 C 若偵測到管理角色 →**403**「請使用 AD、Email 驗證或本地緊急登入」。
- **例外**：`is_protected=true` 之 break-glass 帳號僅走路徑 B，不走路徑 C。

#### AD 帳號（username）輸入規範（§5.1.1）

`sAMAccountName` 須通過**白名單**（前後端一致），防 LDAP injection 與異常輸入：

| 規則 | 說明 |
| ---- | ---- |
| 格式 | `^[a-zA-Z0-9][a-zA-Z0-9._-]{0,19}$`（1～20 字元） |
| 禁止 | 空白、`\`、`/`、`@`、引號、LDAP 特殊字元、SQL 拼接 |
| 正規化 | trim 後轉小寫 |
| 密碼 | 最長 128；不記 log、不寫 DB |
| 查詢 | SQLAlchemy 參數化；LDAP filter 使用 `escape_filter_chars` |

#### JIT 自動建檔（免對照表）

AD 帳號（sAMAccountName）與 emp_id 不同，但因 IT_Admin **不參與訓練**，無需對映到 HR 員編；直接以 AD 帳號自動建立「管理帳號」即可。

| 步驟 | 行為 |
| ---- | ---- |
| 1 | AD 驗證成功且屬 IT_Admin，取得 `sAMAccountName`（如 `it01`）、`displayName`、**`mail`** |
| 2 | 查 `users.emp_id = 'it01'`（管理帳號 emp_id 即 ad_username） |
| 3a | **存在** → 更新 `name`、`ad_username`、**`email`**、`email_verified_at`、`last_login_at`、確保掛管理角色與 `is_trainee=false` |
| 3b | **不存在** → **自動建立**：`emp_id=ad_username`、`name=displayName`、`email=mail`、`auth_source=ad`、`is_trainee=false`、掛 `AD_ADMIN_ROLE_NAME` 角色 |
| 4 | 以該 `emp_id` 簽發 JWT |

**emp_id 撞號保護**：AD 帳號通常為英數（如 `it01`），與純數字員編天然區隔；若 `ad_username` 恰為純數字且與既有員編相同，建檔時應加前綴或拒絕並告警，避免覆蓋真實員工。

> **無需對照表**：人員異動由 AD 群組控管 —— 移出 `IT_Admin` 後群組檢查（步驟 3）即 403，帳號自動失效；停用之 JIT 管理帳號可保留供稽核或定期清理。

#### 情境範例（JIT 自動建檔）

以下以 AD 帳號 `it01`（顯示名稱「王小明」）為例，說明各情境之系統行為。

**情境 1：新進 IT 首次登入（本地尚無帳號）**

1. IT 主管在 AD 將 `it01` 加入 `IT_Admin` 群組。
2. 王小明於管理端選「AD 登入」，輸入 `it01` / AD 密碼。
3. 後端 LDAPS bind 成功 → 查 `memberOf` 含 `IT_Admin` → 通過。
4. 查 `users.emp_id='it01'`：**不存在** → 自動建立：`emp_id=it01`、`name=王小明`、`email`（自 AD mail）、`auth_source=ad`、`is_trainee=false`、掛 `系統管理` 角色。
5. 簽發 JWT，登入管理端。**全程未維護任何對照表。**

**情境 2：既有 IT 再次登入**

1. 王小明再次以 `it01` 登入，群組檢查通過。
2. 查 `users.emp_id='it01'`：**已存在** → 僅更新 `name`、`last_login_at`，確認角色與 `is_trainee=false`。
3. 不重複建檔，簽發 JWT。

**情境 3：IT 離職／調職（移出群組）**

1. IT 主管在 AD 將 `it01` 移出 `IT_Admin`。
2. 王小明下次登入：bind 可能仍成功（AD 帳號未停用），但 `memberOf` **不含** `IT_Admin` → **403**。
3. 系統無需任何異動；本地殘留的 `it01` 帳號因無法再通過群組檢查而自然失效，可留存稽核或定期清理。

**情境 4：JIT 管理帳號不應出現在訓練流程**

1. 管理員建立「全員訓練計畫」。
2. 考試中心、受課對象挑選、報到總覽、成績統計查詢時，因 `it01` 之 `is_trainee=false` 一律被排除。
3. 結果：`it01` 不被列為應考、不灌水應到人數、不出現在成績報表。

**情境 5：AD 故障（Email OTP + break-glass）**

1. DC 不可達，`/auth/login/admin` 回 **503**。
2. `it01` 曾於 30 天內 AD 登入成功且本地有 `email` → 改走 **路徑 D**：請求 OTP → 收信 → 驗證成功 → `auth_src=email_fallback`。
3. 若無 JIT 紀錄、無 email、或 OTP 失敗 → 改用 **路徑 B** break-glass `admin` 登入。

**情境 6：AD 正常時不得走 Email OTP**

1. DC 可連線時呼叫 `/auth/login/admin/email/request` → **400**「請使用 AD 登入」。

### 5.2 設定環境變數化（新增 `backend/app/config.py`）

以 `pydantic-settings` 集中讀取，**禁止硬編碼**：

| 變數                      | 用途                                  | 範例                                   |
| ------------------------- | ------------------------------------- | -------------------------------------- |
| `JWT_SECRET_KEY`        | 取代 `auth_utils.SECRET_KEY` 硬編碼 | `openssl rand -hex 32` 產生          |
| `AD_ENABLED`            | AD 登入總開關                         | `true`                               |
| `AD_SERVER_URI`         | DC 位址                               | `ldaps://dc.yourco.local:636`        |
| `AD_BASE_DN`            | 搜尋起點                              | `DC=yourco,DC=local`                 |
| `AD_DOMAIN`             | UPN／網域                             | `yourco`（組 `user@yourco.local`） |
| `AD_ADMIN_GROUP`        | 准入群組 CN                           | `IT_Admin`                           |
| `AD_ADMIN_ROLE_NAME`    | 對應本地角色名稱                      | `系統管理`                           |
| `LOGIN_BYPASS_ENABLED`  | 驗證碼後門 `0000` 開關（預設關閉）  | `false`                              |
| `PASSWORD_MIN_LENGTH`   | 本地 ADMIN 密碼最短長度               | `12`                                 |
| `PASSWORD_MAX_AGE_DAYS` | 密碼最長有效天數                      | `90`                                 |

### 5.3 AD 驗證服務（新增 `backend/app/services/ad_auth.py`）

- 套件：`ldap3`（需加入`requirements.txt`）。
- 函式：
  - `authenticate_ad(username, password) -> dict | None`：LDAPS bind；成功回`{ad_username, display_name, groups, mail}`。
  - `is_in_admin_group(groups) -> bool`：比對`AD_ADMIN_GROUP`（以`CN=IT_Admin,` 前綴判斷，忽略大小寫）。
- 連線失敗拋連線錯誤（端點轉 **503**，供路徑 D 分流）；帳密錯誤、查無使用者回 `None`（401）。
- **不負責**本地建檔；JIT upsert 由登入端點處理（§5.1 JIT 自動建檔）。

### 5.3.3 Email OTP 備援服務（新增 `backend/app/services/email_otp.py`、`smtp_mailer.py`）

- **僅 AD 不可達時**允許請求 OTP；AD 正常 → 400。
- OTP **雜湊**存入 `admin_login_otps` 表，不存明文。
- 寄信經 SMTP（環境變數）；失敗 → 503。
- 資格：`is_trainee=false`、有管理角色、`email` 非空、`last_login_at` 在時間窗內、曾 `auth_source=ad`。
- 詳見技術設計 §5.4。

### 5.3.2 JIT 管理帳號隔離（排除於訓練／考試）

因 IT_Admin 為純管理身分，JIT 建立之帳號（`is_trainee=false`）須於下列查詢一律排除，避免污染訓練資料：

| 模組 | 落點（現況） | 處理 |
|------|--------------|------|
| 考試中心計畫列表 | `exam_center.py`「全員計畫（無指定對象）」會列給所有人 | 過濾 `is_trainee=false` 帳號，不列為應考 |
| 受課對象挑選 | 訓練計畫編輯之部門／個人選人清單 | 不顯示 JIT 管理帳號 |
| 報到總覽 / 應到統計 | 以 `users` 為母體 | 排除 JIT 管理帳號 |
| 成績 / 報表統計 | 部門成員、成績清單 | 排除 JIT 管理帳號 |

### 5.3.1 密碼政策服務（新增 `backend/app/services/password_policy.py`）

- `validate_password_complexity(password) -> None`：不符則拋錯。
  - 長度 ≥`PASSWORD_MIN_LENGTH`（12）
  - 含小寫、大寫、數字、特殊符號各至少一個
- `is_password_expired(password_changed_at) -> bool`：超過`PASSWORD_MAX_AGE_DAYS`（90）為 true。
- **不實作**密碼歷史比對。
- 雜湊：`auth_utils.hash_password` / `verify_password`（**bcrypt 套件直接**；勿用 `passlib` + bcrypt 5.x，見 Cloud Code 提示詞 §10）。

### 5.4 新增端點

#### 5.4.1 `POST /api/auth/login/admin`（AD 登入）

- Request：`{ username, password }`（username = AD sAMAccountName，須符合 §5.1.1）。
- 行為：依 §5.1 路徑 A；JIT upsert 管理帳號後簽 JWT，`auth_src=ad`。
- `AD_ENABLED=false` →**503**（不影響路徑 B、D 前置條件）。
- DC 連線失敗 → **503** + 提示可改 Email OTP（路徑 D）。

#### 5.4.2 `POST /api/auth/login/admin/email/request`（Email OTP 請求）

- Request：`{ username }`（§5.1.1 白名單）。
- 前置：AD 不可達 + `AD_FALLBACK_EMAIL_ENABLED`；否則 **400**。
- 成功：200 + `{ masked_email, expires_in_seconds }`；頻率超限 → **429**；無資格 → **403**。

#### 5.4.3 `POST /api/auth/login/admin/email/verify`（Email OTP 驗證）

- Request：`{ username, otp_code }`（6 位數字）。
- 成功：簽 JWT，`auth_src=email_fallback`；失敗 → **401**。

#### 5.4.4 `POST /api/auth/login/local`（本地 break-glass）

- Request：`{ emp_id, password }`。
- 僅允許`is_protected=true` 且具管理角色之帳號（預設`BREAK_GLASS_EMP_ID=admin`）。
- 失敗計數：連續**6 次** → 設定`locked_until = now + 15 分鐘`。
- 密碼過期（90 天）：回傳`{ must_change_password: true, change_token: "..." }`，**不發完整管理 JWT**；須呼叫改密 API 後才取得完整 token。
- **不受 `AD_ENABLED` 影響**（AD 全斷仍可登入）。

#### 5.4.5 `POST /api/auth/password/change`（改密）

- Request：`{ change_token 或 已登入 JWT, old_password, new_password }`。
- 新密碼須通過 §5.3.1 複雜度；成功後更新`password_hash`、`password_changed_at`，清除`must_change_password`。
- 改密成功後簽發完整 JWT。

#### 5.4.6 既有 `/login` 調整

- 若使用者角色屬管理端角色 →**403**「請使用 AD 或本地緊急登入」。
- 一般`User` 角色不受影響。

## 8. 風險與營運規範

| 項目                            | 說明                                                                                              |
| ------------------------------- | ------------------------------------------------------------------------------------------------- |
| UPN 格式                        | 需 IT 確認 bind 用 `user@domain.local` 或 `DOMAIN\user`                                       |
| 群組巢狀                        | IT_Admin 若為巢狀群組，`memberOf` 可能不含；必要時用 `LDAP_MATCHING_RULE_IN_CHAIN`            |
| 憑證信任                        | LDAPS 需信任 DC 憑證                                                                              |
| **免對照表（已採 JIT）** | 人員異動由 AD `IT_Admin` 群組單一來源控管；移出群組者下次登入即 403，**無需維護對照表**   |
| **JIT 帳號隔離**          | 管理帳號 `is_trainee=false`，須確實排除於考試／受課／報到／統計（見 §5.3.2），否則污染訓練資料 |
| **emp_id 撞號**          | `ad_username` 若為純數字且與既有員編相同，建檔須加前綴或拒絕，避免覆蓋真實員工                  |
| **break-glass（已保留）** | 單一本地 ADMIN（`is_protected`）；**最後手段**；密碼由資安／IT 保管 |
| AD 故障                         | **優先**路徑 D（Email OTP，須曾 AD 登入且有 email）；**其次**路徑 B break-glass |
| **Email OTP 殘留權限**    | 斷線期間無法查 AD 群組；以 30 天時間窗 + 本地 JIT 資格控管；AD 恢復後應改回路徑 A |
| **AD 無 mail**            | 該 IT 帳號無法走路徑 D；僅 break-glass 或等 AD 恢復                         |
| **LDAP／輸入注入**        | username 白名單 + LDAP escape；SQL 參數化查詢                              |
| **OTP 濫發**              | 頻率限制 + 雜湊儲存 + 稽核 log                                             |
| Security Review           | 實作完成後必做；無未修復 High 才上線（技術設計 §11）                          |
| 鎖定                            | 6 次失敗鎖 15 分鐘；避免唯一 break-glass 被暴力破解（仍須強密碼）                                 |
| 密碼到期                        | 強制改密才放行，避免鎖死唯一保險絲                                                                |
| 離職 IT 清理                    | 移出群組即無法登入；殘留之停用 JIT 帳號可定期清理（不影響訓練資料）                               |
