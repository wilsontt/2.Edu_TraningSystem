【目標】用 Locust 對本機後端做壓力測（禁止打生產）。

場景：
1. 同時 100 名員工登入（路徑 C：POST /api/auth/login；模擬 QR 掃碼後的員工登入，不是 AD）
2. 每人依序對兩個計畫考試並交卷：
   - 「AI天然吔～尚好」
   - 「2026新人計劃」（若 DB 標題略有不同，先查 training_plans 確認）
3. 作答隨機；約 40% 使用者盡量答對（可從 DB 讀正確答案，僅測試環境）、60% 亂填，產生及格/不及格混雜

技術約束：
- 直打後端 http://127.0.0.1:8000（不要經 Vite /training 代理）
- 測試環境必須 LOGIN_BYPASS_ENABLED=true，驗證碼用 0000
- 開考/交卷前必須 POST /api/exam/plan/{plan_id}/attendance/checkin
- API：
  - GET /api/auth/captcha
  - POST /api/auth/login { emp_id, captcha_id, answer }
  - POST /api/exam/plan/{plan_id}/attendance/checkin  (Bearer)
  - GET /api/exam/start/{plan_id}  (Bearer)
  - POST /api/exam/submit/{plan_id}  { "answers": { <question_id>: "<option>" } }  (Bearer)
  - answers 的 key 在 Pydantic 是 dict[int,str]，JSON 會變字串 key，後端兩者都接
- SQLite 可能鎖庫：腳本與報告要標註這點；spawn-rate 不要一次炸太猛（例如 10～20/s）

請產出：
1. tests/load/README.md（前置、執行指令、解讀指標）
2. tests/load/prepare_users.py（建立/確認 100 名測試員工、兩計畫受課範圍；可重複執行）
3. tests/load/locustfile.py（HttpUser；on_start 登入；task 報到→開考→作答→交卷×2 計畫）
4. tests/load/run.sh（例：--users 100 --spawn-rate 20 --run-time 10m --headless）
5. 不要改業務程式碼，除非為了測試必須且先說明
6. 跑完後輸出：失敗率、p50/p95 latency、常見錯誤狀態碼；若 SQLite locked 要寫進報告

驗收：
- locust headless 可跑完
- Network/報表可見 login、checkin、start、submit
- DB 內兩計畫有大量 ExamRecord／submit_time
