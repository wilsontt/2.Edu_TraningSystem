## 3. 架構概述

本系統採用 **RBAC（Role-Based Access Control，基於角色的存取控制）** 架構：

```
使用者 (User) 
  └─ 角色 (Role) 
      └─ 系統功能 (SystemFunction) ← 權限代碼 (Code)
```

**核心概念**：
- **使用者**：系統中的每個帳號（以 `emp_id` 為唯一識別）
- **角色**：使用者所屬的權限群組（如 Admin、User）
- **系統功能**：系統中的功能點，每個功能有唯一的 `code`（如 `menu:admin:user`）
- **權限代碼**：用於判斷使用者是否有權限存取特定功能的字串標識

**關係**：
- 一個使用者對應一個角色（多對一）
- 一個角色可擁有多個系統功能（多對多，透過 `role_functions` 關聯表）

## 4. 資料模型

### 4.1 資料表結構

#### User（使用者）
- `emp_id` (String, PK)：員工編號，主鍵
- `name` (String)：姓名
- `dept_id` (Integer, FK)：所屬部門 ID
- `role_id` (Integer, FK)：角色 ID
- `status` (String)：帳號狀態（active/inactive）

#### Role（角色）
- `id` (Integer, PK)：角色 ID
- `name` (String, Unique)：角色名稱（如 "Admin"、"User"）

#### SystemFunction（系統功能）
- `id` (Integer, PK)：功能 ID
- `name` (String)：功能名稱（如 "人員管理"）
- `code` (String, Unique)：權限代碼（如 "menu:admin:user"）
- `parent_id` (Integer, FK, Nullable)：父功能 ID（支援樹狀結構）
- `path` (String, Nullable)：前端路由路徑（如 "/admin/users"）

#### role_functions（角色與功能關聯表）
- `role_id` (Integer, FK)：角色 ID
- `function_id` (Integer, FK)：功能 ID

### 4.2 關聯關係

```
User (1) ──→ (N) Role
              │
              │ (多對多)
              ↓
         SystemFunction
```

**SQLAlchemy 關聯定義**：
```python
# User
role = relationship("Role", back_populates="users")

# Role
users = relationship("User", back_populates="role")
functions = relationship("SystemFunction", secondary=role_functions, back_populates="roles")

# SystemFunction
roles = relationship("Role", secondary=role_functions, back_populates="functions")
children = relationship("SystemFunction", backref=backref("parent", remote_side=[id]))
```

## 5. 認證機制

### 5.1 JWT Token 認證

**認證流程**：
1. 使用者登入（`POST /api/auth/login`）
2. 後端驗證使用者存在且狀態為 active
3. 後端簽發 JWT Token（包含 `emp_id` 和 `role`）
4. 前端儲存 Token 至 `localStorage`
5. 後續請求在 Header 中帶上 `Authorization: Bearer {token}`

**Token 內容**：
```json
{
  "sub": "emp_id",
  "role": "Admin",
  "exp": 1234567890
}
```

**Token 設定**（`backend/app/auth_utils.py`）：
- 演算法：HS256
- 過期時間：480 分鐘（8 小時）
- 密鑰：`SECRET_KEY`（開發環境使用固定值，生產環境應使用環境變數）

### 5.2 取得當前使用者

**依賴注入**：`get_current_user`（`backend/app/routers/auth.py`）

```python
async def get_current_user(token: str = Depends(oauth2_scheme), db: Session = Depends(get_db)):
    # 1. 驗證 Token
    payload = auth_utils.verify_token(token)
    if not payload:
        raise HTTPException(status_code=401, detail="無效的認證憑證")
    
    # 2. 從資料庫載入使用者（預載入 role 和 functions）
    user = db.query(models.User).options(
        joinedload(models.User.role).joinedload(models.Role.functions)
    ).filter(models.User.emp_id == emp_id).first()
    
    return user
```

**登入回應**（`POST /api/auth/login`）：
```json
{
  "access_token": "eyJ...",
  "token_type": "bearer",
  "user": {
    "emp_id": "12345",
    "name": "張三",
    "dept_name": "IT部",
    "role": "Admin",
    "functions": ["menu:home", "menu:admin", "menu:admin:user", ...]
  }
}
```

## 6. 權限檢查機制

### 6.1 權限檢查函數

**位置**：`backend/app/routers/auth.py` 之 `check_permission`

**實作邏輯**：
```python
def check_permission(required_func_code: str):
    async def permission_dependency(current_user: models.User = Depends(get_current_user)):
        # 1. 確保使用者有角色
        if not current_user.role:
            raise HTTPException(status_code=403, detail="使用者未分配角色")
        
        # 2. Admin 或 System Admin 擁有所有權限（超級使用者）
        role_name = current_user.role.name
        if role_name in ["Admin", "System Admin"]:
            return current_user
        
        # 3. 檢查角色的功能清單中是否包含要求的代碼
        if any(func.code == required_func_code for func in current_user.role.functions):
            return current_user
        
        # 4. 權限不足
        raise HTTPException(status_code=403, detail="權限不足，無法存取此功能")
    
    return Depends(permission_dependency)
```

**使用方式**：
```python
@router.get("/users")
def get_users(
    db: Session = Depends(get_db),
    current_user = check_permission("menu:admin:user")  # 需要此權限
):
    # ...
```

### 6.2 權限代碼命名規範

**格式**：`{類型}:{模組}:{功能}`

**範例**：
- `menu:home`：首頁選單
- `menu:plan`：訓練計畫選單
- `menu:admin`：系統管理選單（父功能）
- `menu:admin:user`：人員管理子功能
- `menu:admin:role`：角色管理子功能
- `menu:admin:perm`：權限管理子功能
- `menu:admin:dept`：單位管理子功能
- `menu:admin:func`：功能清單管理子功能

## 8. API 端點

### 8.1 認證相關

| 端點 | 方法 | 說明 | 權限要求 |
|------|------|------|----------|
| `/api/auth/login` | POST | 使用者登入 | 無 |
| `/api/auth/register` | POST | 使用者註冊 | 無 |
| `/api/auth/me` | GET | 取得當前使用者資訊 | 需 Token |

### 8.2 角色管理

| 端點 | 方法 | 說明 | 權限要求 |
|------|------|------|----------|
| `/api/admin/roles` | GET | 取得所有角色 | `menu:admin:role` |
| `/api/admin/roles` | POST | 新增角色 | `menu:admin:role` |
| `/api/admin/roles/{role_id}` | PUT | 更新角色名稱 | `menu:admin:role` |
| `/api/admin/roles/{role_id}` | DELETE | 刪除角色 | `menu:admin:role` |

**保護機制**：
- Admin 角色無法更名
- Admin 角色無法刪除
- 有成員的角色無法刪除
- 有權限設定的角色無法刪除（需先清空權限）

### 8.3 權限管理

| 端點 | 方法 | 說明 | 權限要求 |
|------|------|------|----------|
| `/api/admin/roles/{role_id}/permissions` | GET | 取得角色的功能 ID 列表 | `menu:admin:perm` |
| `/api/admin/roles/{role_id}/permissions` | PUT | 更新角色權限 | `menu:admin:perm` |

**保護機制**：
- Admin 角色的權限無法變更

### 8.4 功能清單管理

| 端點 | 方法 | 說明 | 權限要求 |
|------|------|------|----------|
| `/api/admin/functions` | GET | 取得所有系統功能（樹狀結構） | `menu:admin` |
| `/api/admin/functions` | POST | 新增系統功能 | `menu:admin:func` |
| `/api/admin/functions/{id}` | PUT | 更新系統功能 | `menu:admin:func` |
| `/api/admin/functions/{id}` | DELETE | 刪除系統功能 | `menu:admin:func` |

**保護機制**：
- 有子節點的功能無法刪除
- 已被角色引用的功能無法刪除（需先移除角色權限）

### 8.5 人員管理

| 端點 | 方法 | 說明 | 權限要求 |
|------|------|------|----------|
| `/api/admin/users` | GET | 取得所有使用者 | `menu:admin:user` |
| `/api/admin/users/{emp_id}` | PUT | 更新使用者（角色、單位、狀態） | `menu:admin:user` |

**保護機制**：
- `admin` 帳號的角色與單位不可變更
- `admin` 帳號無法被停用

## 9. 保護機制

### 9.1 Admin 角色保護

**後端檢查**（`backend/app/routers/admin.py`）：

1. **角色更名**：
```python
if db_role.name == "Admin":
    raise HTTPException(status_code=400, detail="系統管理員角色 (Admin) 無法更名")
```

2. **角色刪除**：
```python
if db_role.name == "Admin":
    raise HTTPException(status_code=400, detail="無法刪除系統管理員角色")
```

3. **權限變更**：
```python
if role.name == 'Admin':
    raise HTTPException(status_code=400, detail="系統管理者 (Admin) 的權限無法變更")
```

### 9.2 Admin 使用者保護

**後端檢查**（`backend/app/routers/admin.py`）：

1. **角色變更**：
```python
if emp_id.lower() == 'admin':
    if user_update.role_id is not None and user_update.role_id != db_user.role_id:
        raise HTTPException(status_code=400, detail="系統預設管理員的角色不能變更")
```

2. **部門變更**：
```python
if emp_id.lower() == 'admin':
    if user_update.dept_id is not None and user_update.dept_id != db_user.dept_id:
        raise HTTPException(status_code=400, detail="系統預設管理員的部門不能變更")
```

3. **帳號停用**：
```python
if emp_id.lower() == 'admin' and user_update.status != 'active':
    raise HTTPException(status_code=400, detail="系統預設管理員不能被停用")
```

### 9.3 前端保護

**權限管理介面**（`frontend/src/components/admin/PermissionManager.tsx`）：

## 10. 初始化流程

### 10.1 資料庫初始化

**腳本**：`backend/app/init_db.py`

**初始化步驟**：

1. **建立基礎角色**：
   - `Admin`：系統管理員
   - `User`：一般使用者

2. **建立基礎功能選單**：
   - 根功能：`menu:home`、`menu:plan`、`menu:exam`、`menu:report`、`menu:admin`
   - 系統管理子功能：`menu:admin:dept`、`menu:admin:user`、`menu:admin:role`、`menu:admin:perm`、`menu:admin:func`

3. **配置角色權限**：
   - `Admin` 角色：擁有所有功能
   - `User` 角色：僅擁有 `menu:home` 和 `menu:report`

4. **建立預設管理員帳號**：
   - `emp_id`: "admin"
   - `name`: "系統管理員"
   - `role_id`: Admin 角色 ID
   - `dept_id`: IT部 ID
   - `status`: "active"

**執行方式**：
```bash
cd backend
python -m app.init_db
```

## 12. 設計原則

### 12.1 最小權限原則

- 預設情況下，新註冊的使用者為 `User` 角色，僅擁有基本權限
- 需由管理員明確指派權限，而非預設擁有所有權限

### 12.2 分層保護

- **前端**：使用者體驗優化，隱藏無權限的功能
- **後端**：真正的權限檢查，確保資料安全

### 12.3 可擴展性

- 系統功能採用樹狀結構，支援父子關係
- 權限代碼採用命名規範，易於管理與擴充

### 12.4 安全性

- Admin 角色與 admin 使用者受多重保護，防止誤操作
- 所有 API 端點皆需權限驗證
- JWT Token 有過期時間，降低安全風險
