## 📋 事件摘要

系統在修復登入功能時，意外導致資料庫資料全部遺失，包括：
- 所有用戶帳號（9個用戶）
- 訓練計劃資料
- 考試工坊題目
- 考試中心記錄
- 成績中心資料

經過緊急處理，已從備份資料庫恢復大部分資料，但部分資料因表結構差異仍需進一步處理。

## 🔍 問題發現過程

### 階段一：登入功能錯誤

#### 1.1 初始問題
**時間**: 2025-01-09 上午  
**症狀**:
- 前端無法登入系統
- 瀏覽器控制台顯示兩個錯誤：
  1. **CORS 錯誤**: `Access to XMLHttpRequest at 'http://192.168.0.104:8000/api/auth/login' from origin 'http://192.168.0.104:5173' has been blocked by CORS policy: No 'Access-Control-Allow-Origin' header is present on the requested resource.`
  2. **500 內部伺服器錯誤**: `POST http://192.168.0.104:8000/api/auth/login net::ERR_FAILED 500 (Internal Server Error)`

**檢查方式**:
- 查看瀏覽器開發者工具控制台
- 檢查後端終端錯誤訊息
- 檢查後端程式碼（`backend/app/main.py` 和 `backend/app/routers/auth.py`）

**發現的問題**:
1. CORS 設定使用 `allow_origin_regex="https?://.*"`，無法正確匹配本地 IP 地址格式
2. 登入端點在查詢用戶時沒有預載入關聯資料（`department`、`role.functions`），導致 lazy loading 錯誤

#### 1.2 第一次修復

**修復內容**:

1. **CORS 設定修復** (`backend/app/main.py`):
   ```python
   # 修改前
   allow_origin_regex="https?://.*"
   
   # 修改後
   allow_origins=["*"]  # 開發環境允許所有來源
   ```

2. **登入端點修復** (`backend/app/routers/auth.py`):
   ```python
   # 添加 joinedload 預載入關聯資料
   user = db.query(models.User).options(
       joinedload(models.User.department),
       joinedload(models.User.role).joinedload(models.Role.functions)
   ).filter(models.User.emp_id == req.emp_id).first()
   ```

3. **添加調試資訊**:
   - 在登入端點添加詳細的調試輸出
   - 在前端添加請求日誌

**結果**: CORS 錯誤解決，但出現新的錯誤

**症狀**:

   ```bash

   ```

**結果**: 資料庫表已創建，但發現所有資料遺失

### 階段三：資料遺失問題

#### 3.1 問題發現
**時間**: 2025-01-09 下午  
**症狀**:
- 用戶可以登入，但所有帳號都不見了
- 登入後發現訓練計劃、考試工坊、考試中心、成績中心所有資料都是空的

**檢查方式**:
```bash
# 檢查當前資料庫用戶數
sqlite3 ../data/education_training.db "SELECT COUNT(*) FROM users"

# 檢查備份資料庫用戶數
sqlite3 "../data/education_training 2.db" "SELECT COUNT(*) FROM users"
```

**檢查結果**:
- 當前資料庫: 0 個用戶
- 備份資料庫: 9 個用戶
- 確認所有資料都已遺失

#### 3.2 原因分析

**根本原因**:
1. **自動初始化邏輯問題**: 
   - `Base.metadata.create_all(bind=engine)` 雖然理論上只會創建不存在的表，不會刪除資料
   - 但可能因為以下原因導致資料遺失：
     - 資料庫檔案被重新創建
     - 執行 `init_db()` 時有問題
     - 資料庫連接路徑錯誤，連接到新的空資料庫

2. **資料庫初始化腳本問題**:
   - `init_db.py` 中的 `Base.metadata.create_all()` 可能在某些情況下會重置資料庫
   - 沒有檢查資料庫是否已有資料的保護機制

3. **缺乏資料備份機制**:
   - 沒有自動備份機制
   - 只有手動備份檔案 `education_training 2.db`

#### 2. 使用 SQLite 直接恢復
```bash

SELECT '恢復完成，用戶數:' || COUNT(*) FROM users;
DETACH DATABASE backup;
SQL
```

## 📝 經驗教訓

### 1. 資料庫操作風險
- ⚠️ **自動初始化要謹慎**: 不應該在每次啟動時都執行 `create_all()`
- ⚠️ **要有資料檢查**: 執行初始化前應該檢查是否已有資料
- ⚠️ **要有備份機制**: 重要操作前應該自動備份

### 2. 表結構變更管理
- ⚠️ **版本控制**: 表結構變更應該有版本記錄
- ⚠️ **遷移腳本**: 應該使用遷移腳本而不是直接修改表結構
- ⚠️ **向下相容**: 新欄位應該允許 NULL 或設置預設值

### 3. 錯誤處理
- ✅ **調試資訊**: 添加詳細的調試資訊有助於快速定位問題
- ✅ **錯誤訊息**: 清晰的錯誤訊息可以幫助理解問題
- ✅ **日誌記錄**: 重要操作應該記錄日誌
