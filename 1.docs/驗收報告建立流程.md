## 📋 文件說明

本文件說明任務完成後建立驗收報告的標準流程，確保符合專案憲章 **P0（SDD Phase 5：驗收）** 與 **P3（清晰與可測試性）** 之要求，以及 ISO 27001:2022 之文件化與可追溯性需求。

## ✅ 本專案驗收清單（任務完成時快速對照）

任務完成時，依序勾選以下項目：

- [ ] **1. 驗收條件檢查**：所有驗收條件逐項檢查並標註 ✅/❌，未通過者記錄原因與後續處理
- [ ] **2. 測試驗證**：功能測試／整合測試已執行並記錄結果，測試通過
- [ ] **3. 建立驗收報告**：依命名規則建立報告檔，存放於 `1.docs/系統建置與驗證報告/`，且包含所有必要章節
- [ ] **4. 更新任務狀態**：更新 `0.standards/1.綠地專案文件/3.tasks.md` 任務狀態為完成並附驗收報告連結
- [ ] **5. 更新驗收報告總覽**：更新 `1.docs/系統建置與驗證報告/README.md`，標記任務完成並附連結
- [ ] **6. 階段完成檢查**（若該階段任務全數完成）：建立「階段完成總結」並更新 README

## 📝 任務完成檢查清單（詳細）

### 1. 驗收條件檢查
- [ ] 確認所有驗收條件都已達成
- [ ] 逐項檢查驗收條件，標註通過（✅）或未通過（❌）
- [ ] 如有未通過項目，記錄原因與後續處理方式

### 2. 測試驗證
- [ ] 執行功能測試
- [ ] 執行整合測試（如適用）
- [ ] 記錄測試結果與測試覆蓋率
- [ ] 確認所有測試通過

### 3. 建立驗收報告
- [ ] 建立驗收報告檔案
- [ ] 報告命名格式：`{任務編號}-{任務名稱}-驗收報告.md`
- [ ] 報告存放位置：`1.docs/系統建置與驗證報告/階段{N}-{階段名稱}/`
- [ ] 報告包含所有必要章節（見下方「驗收報告內容要求」）

### 4. 更新任務狀態
- [ ] 更新 `0.standards/1.綠地專案文件/3.tasks.md` 中的任務狀態為已完成
- [ ] 在任務描述中標註 ✅ 符號
- [ ] 新增驗收報告連結
- [ ] 更新任務統計資訊（如適用）

### 5. 更新驗收報告總覽
- [ ] 更新 `1.docs/系統建置與驗證報告/README.md`
- [ ] 在「已完成任務」章節中標記任務為已完成（✅）
- [ ] 新增驗收報告連結
- [ ] 更新「最後更新」日期

### 6. 階段完成檢查（如適用）
- [ ] 檢查階段內所有任務是否都已完成
- [ ] 如階段完成，建立「階段完成總結」文件
- [ ] 更新階段完成總結到 README.md

## 📝 驗收報告內容要求

每個驗收報告**必須**包含以下章節：

### 1. 任務概述
- 任務編號
- 任務名稱
- 執行日期
- 執行人員
- 狀態（已完成/未完成）

### 2. 執行內容
- 詳細的實作內容
- 使用的技術與工具
- 實作過程中的關鍵決策

### 3. 驗收條件檢查
- 逐項列出所有驗收條件
- 標註通過（✅）或未通過（❌）
- 如有未通過項目，說明原因與處理方式

### 4. 測試結果
- 測試執行結果
- 測試覆蓋率（如適用）
- 測試案例說明
- 測試通過/失敗情況

### 5. 交付成果
- 完成的工作項目清單
- 建立的檔案與目錄
- 修改的檔案清單
- 相關文件連結

### 6. 相關文件
- 相關的設計文件連結
- 相關的任務文件連結
- 相關的需求文件連結

### 7. 備註
- 其他說明事項
- 遇到的問題與解決方案
- 後續改進建議

### 8. 驗收狀態
- 驗收結果（通過/未通過）
- 驗收日期
- 驗收人員

## 📁 驗收報告檔案命名規則

**格式**：`{任務編號}-{任務名稱}-驗收報告.md`

**範例**：
- `T-1-1-建立後端專案結構-驗收報告.md`
- `T-2-2-實作員工編號登入與三欄式註冊-驗收報告.md`
- `T-3-2-實作大項目與細項目兩層級管理-驗收報告.md`

**注意事項**：
- 任務編號可與 `3.tasks.md` 一致（如 T1.1、T2.2、T3.2）
- 任務名稱使用繁體中文
- 檔案名稱不包含特殊字元（如 `/`, `\`, `:`, `*`, `?`, `"`, `<`, `>`, `|`）
- 使用連字號 `-` 分隔

## 🔗 文件追溯性要求

驗收報告**必須**可追溯至：

1. **3.tasks.md**
   - 路徑：`0.standards/1.綠地專案文件/3.tasks.md`
   - 在任務描述中新增驗收報告連結
   - 更新任務狀態為已完成

2. **plan / spec**（如存在）
   - 路徑：`0.standards/1.綠地專案文件/` 下之計畫或規格
   - 在相關設計章節中引用驗收報告

3. **1.docs/plans/**
   - 若該任務有對應實作計劃，在計劃中可引用驗收報告

## 📊 階段完成總結要求

當一個階段的所有任務完成後，**必須**建立「階段完成總結」文件：

### 文件命名
- **格式**：`階段{N}-{階段名稱}-完成總結.md`
- **範例**：`階段2-認證與功能權限系統-完成總結.md`

### 文件存放位置
- `1.docs/系統建置與驗證報告/階段{N}-{階段名稱}/`

### 文件內容要求
1. **階段概述**：階段目標、對應文件、預估時程、里程碑  
2. **完成任務清單**：按工作項目分類，列出所有已完成任務  
3. **交付成果總結**：階段完成的主要交付成果  
4. **測試結果總結**：整體測試結果、覆蓋率、通過率  
5. **遇到的問題與解決方案**  
6. **後續改進建議**  
7. **階段總結**：完成狀態、達成目標評估、對專案整體進度的貢獻  

### 建立時機
- 階段內所有任務的驗收報告都完成後立即建立  
- **必須**在階段完成後 3 個工作天內建立  

---

## ⚠️ 重要提醒

1. **驗收報告必須在任務完成後立即建立**  
   - 不得延遲建立驗收報告  
   - 驗收報告與任務完成同步  

2. **所有驗收條件必須達成**  
   - 如有未達成的驗收條件，必須記錄原因  
   - 未達成的驗收條件必須有後續處理計畫  

3. **文件必須保持一致性**  
   - `3.tasks.md` 中的任務狀態必須與驗收報告一致  
   - `1.docs/系統建置與驗證報告/README.md` 中的已完成任務清單必須與實際狀態一致  

4. **符合專案憲章與 ISO 27001:2022**  
   - 完整的文件記錄  
   - 可追溯性  
   - 驗收條件明確  
   - 測試結果記錄
