## 一、目的

彙整本機開發時，**前端（npm / Vite）**與**後端（Python / `.venv` / Uvicorn）**常見異常、故障與無法重新安裝等問題之**判斷方式與修復步驟**，供開發者與維運人員快速排除，無須翻查零散對話紀錄。

---

## 二、範圍

| 包含 | 不包含 |
|------|--------|
| macOS / Linux / Windows 本機開發環境 | 正式環境 Docker 部署（見 [生產部署指南](生產部署指南.md)） |
| npm、`node_modules`、Vite dev server | 業務功能 bug（見 `02-棕地專案/規格與計畫/4.修復記錄.md`） |
| Python 虛擬環境、pip、Uvicorn 啟動 | NAS／SMB 連線（見 [NAS與路徑跨平台慣例](NAS與路徑跨平台慣例.md)） |
| 前後端連線（proxy `ECONNREFUSED`） | Cursor IDE 版面問題 |

## 三、權責

| 角色 | 責任 |
|------|------|
| 開發者 | 依本文件自行排除本機環境問題；修復後若為可重現之環境陷阱，應回報以更新本文件 |
| 維運／管理員 | 協助確認 Node、Python、網路與埠占用；不修改版控內 `.venv`（本機產物） |

## 四、名詞解釋

| 名詞 | 說明 |
|------|------|
| **`uv_cwd`** | Node.js / npm 讀取「目前工作目錄」失敗；通常代表 **terminal 的 cwd 已失效**（目錄被刪除、移動或 session 過期） |
| **幽靈 cwd** | Terminal 提示字仍顯示某路徑，但 OS 已無法解析該目錄 |
| **損壞的 `.venv`** | 虛擬環境內 Python 的 `sys.prefix` 異常（如 `/install`），或找不到 `encodings` 等標準函式庫 |
| **uv standalone Python** | 由 [uv](https://github.com/astral-sh/uv) 管理的 Python，prefix 常為 `/install`；**不建議**直接 `python3 -m venv` 建 venv |
| **proxy ECONNREFUSED** | Vite 將 `/training/api/*` 轉發至 `localhost:8000` 時，**後端未啟動**或埠不符 |

## 五、故障對照與修復步驟

### 5.1 前端：npm 無法執行（`uv_cwd`）

#### 症狀

```text
Error: ENOENT: no such file or directory, uv_cwd
  syscall: 'uv_cwd'
```

`npm install`、`npm run dev` 在**同一個已開很久的 terminal** 突然全部失敗；專案目錄其實仍存在。

#### 原因

Terminal 的工作目錄（cwd）已失效。常見觸發：

- 在 terminal 仍開啟時刪除／重建 `frontend/` 或 `node_modules/`
- 專案資料夾被移動或重新命名
- Git 操作清掉目錄後還原
- Cursor / VS Code **舊 terminal session** 未關閉即重開工作區

#### 修復

1. **關閉**出錯的 terminal，**開新 terminal**（不要沿用舊 session）
2. 以**絕對路徑**進入目錄：

```bash
cd "/Users/wilson/5.Projects/3.企業入口網站/2.教育訓練教材及線上考卷/frontend"
pwd    # 必須印出路徑，不可報錯
npm install
npm run dev
```

或在專案根目錄：

```bash
cd "/Users/wilson/5.Projects/3.企業入口網站/2.教育訓練教材及線上考卷"
npm run dev   # 根 package.json 會轉發至 frontend
```

3. 驗證：Vite 顯示 `ready`，網址含 `/training/`（預設 `http://localhost:5173/training/`）

> **注意**：在幽靈 cwd 的 terminal 內執行 `cd frontend` 通常**無法**修復，請開新 terminal。

---

### 5.2 前端：頁面可開但 API 全失敗（`proxy error ECONNREFUSED`）

#### 症狀

Vite 正常，但 terminal 或瀏覽器出現：

```text
[vite] http proxy error: /api/auth/me
AggregateError [ECONNREFUSED]
```

登入、驗證碼、部門列表等請求失敗。

#### 原因

後端 FastAPI **未在 port 8000 運行**。前端 dev proxy 設定見 `frontend/vite.config.ts`：`/training/api/*` → `http://localhost:8000/api/*`。

#### 修復

1. 另開 terminal，依 [§5.3](#53-後端虛擬環境損壞無法啟動) 或 [§5.4](#54-後端正常啟動檢查清單) 啟動後端
2. 確認 `http://localhost:8000/docs` 可開啟
3. 重新整理前端頁面

---

### 5.3 後端：虛擬環境損壞、無法啟動

#### 症狀 A — 找不到 Python

```text
zsh: no such file or directory: .venv/bin/python3
```

#### 症狀 B — 標準函式庫缺失

```text
ModuleNotFoundError: No module named 'encodings'
Fatal Python error: init_fs_encoding: failed to get the Python codec of the filesystem encoding
```

錯誤詳情中常見：

```text
sys.prefix = '/install'
stdlib dir = '/install/lib/python3.12'
sys._base_executable = '.../uv/python/cpython-3.12.12-macos-x86_64-none/...'
```

#### 症狀 C — 在損壞 venv 上重建失敗

```text
python3 -m venv .venv
Error: Command '... ensurepip ...' returned non-zero exit status 1.
```

#### 原因

1. `backend/.venv` **不在版控內**，可能從未建立、被刪除或半殘留
2. 預設 `python3` 指向 **uv standalone Python**（`~/.local/bin/python3`）時，用 `python3 -m venv` 常建出 **prefix 為 `/install` 的損壞 venv**
3. 在**未刪除**舊 `.venv` 的情況下再次 `python3 -m venv .venv`，會疊加更亂的狀態

#### 修復（macOS / Linux，已安裝 uv — 建議）

```bash
cd "/Users/wilson/5.Projects/3.企業入口網站/2.教育訓練教材及線上考卷/backend"

# 1. 必須整個刪除損壞的 venv
rm -rf .venv

# 2. 用 uv 建立（勿用 python3 -m venv）
uv venv .venv --python 3.12
uv pip install -r requirements.txt

# 3. 驗證
.venv/bin/python3 -c "import encodings, uvicorn; print('OK')"

# 4. 啟動
export PYTHONPATH=$PYTHONPATH:.
.venv/bin/python3 -m uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

#### 修復（macOS / Linux，Homebrew Python）

若未安裝 uv，或 uv 建 venv 失敗：

```bash
cd backend
rm -rf .venv

brew install python@3.12   # 若尚未安裝
/opt/homebrew/bin/python3.12 -m venv .venv   # Apple Silicon
# Intel Mac 可能為：/usr/local/bin/python3.12 -m venv .venv

.venv/bin/python3 -m pip install -r requirements.txt
export PYTHONPATH=$PYTHONPATH:.
.venv/bin/python3 -m uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

#### 修復（Windows PowerShell）

```powershell
cd backend
Remove-Item -Recurse -Force .venv -ErrorAction SilentlyContinue

python -m venv .venv
# 若 python 找不到：py -3 -m venv .venv

.\.venv\Scripts\python.exe -m pip install -r requirements.txt
$env:PYTHONPATH = "$PWD;$env:PYTHONPATH"
.\.venv\Scripts\python.exe -m uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

> Windows 請使用 `.\.venv\Scripts\python.exe`，**不可**使用 `.venv/bin/python3`。

### 5.5 前端：依賴異常或需乾淨重裝

#### 症狀

- `npm install` 報錯、套件版本衝突
- `npm run dev` 找不到模組
- 換 Node 版本後行為異常


### 5.6.1 重整後白字「public base URL of /training/」

#### 症狀

- 登入後（或掃 QR 登入後）畫面正常，網址列為 `…/training`（**無**結尾 `/`）
- **重整**後出現：`The server is configured with a public base URL of /training/ - did you mean to visit /training/ instead?`
- 前後端 terminal **無**錯誤 log

#### 原因

Vite `base` 為 `/training/`。React Router `basename="/training"` 導向首頁時，網址列常變成 `/training`（無斜線）。**整頁重新載入**時 Vite 不接受無斜線路徑，回說明文字而非 SPA。此現象與後端無關。

#### 修復（程式已處理）

| 層級 | 作法 |
|------|------|
| Vite dev / preview | `vite.config.ts`：精確路徑 `/training` → 302 `/training/` |
| 前端 | `App.tsx`：掛載時若為 `/training` 則 `replaceState` 成 `/training/` |
| Nginx（正式映像） | `location = /training` → `302 /training/` |

驗證：登入後若網址曾為 `/training`，應自動變成 `/training/`；對無斜線路徑按 F5 應導向後正常進系統。**重啟 Vite**（改 config 後）再測。

### 5.7 考試中心／成績中心空白（schema 未遷移）

#### 症狀

- 登入任何員工編號後，考試中心、成績中心皆無資料
- 瀏覽器 Network：`GET /api/exam/my_exams`、`GET /api/exam/personal/history` 回 **500**
- 後端日誌：`no such column: exam_records.retake_authorized`

#### 原因

程式碼已使用 `exam_records.retake_authorized` 欄位，但本機 `data/education_training.db` 未執行遷移。`init_db` 只建立新表，**不會**修改既有表結構。

#### 修復

```bash
cd backend

# 備份
cp ../data/education_training.db ../data/education_training.db.bak-$(date +%Y%m%d)

# 執行遷移（可重複跑）
.venv/bin/python3 migrations/add_exam_retake_authorization.py
```

重啟後端後重新整理前端。詳見 [MIGRATION_GUIDE.md](資料庫遷移/MIGRATION_GUIDE.md)「已通過授權重考」。

### 5.8 已考過仍跳出「尚未完成報到」

#### 症狀

- 成績詳情顯示已交卷（含多次挑戰）
- 考試中心點「開始考試」→ 彈窗「尚未完成報到」

#### 原因

`exam_records`（考試）與 `attendance_records`（報到）為不同資料表。歷史資料可能在「強制報到檢查」上線前就已交卷，資料庫無對應報到列。

#### 修復

| 情境 | 作法 |
|------|------|
| **本機開發（個人）** | 點「立即報到」即可重考；同一計畫報到過後不會再問 |
| **本機開發（批次）** | `cd backend && .venv/bin/python3 migrations/backfill_attendance_from_exam.py` |
| **Docker 正式環境** | 見 [MIGRATION_GUIDE.md](資料庫遷移/MIGRATION_GUIDE.md)「報到紀錄歷史補齊」 |

### 5.10 Docker：NAS 登入失敗或 SMTP 加密腳本報錯

> **完整步驟**見 [生產部署指南](生產部署指南.md) §5.3～5.7；本節僅列常見根因速查。

#### 症狀 A：`NAS 共享尚未設定（需 SMB_SERVER／SMB_SHARE）`

**根因**：Docker 容器未讀到 SMB 設定。常見誤把設定寫在 `backend/.env`，但映像不含該檔。

**修復**：

1. 編輯 `/opt/apps/enterprise-portal/deploy/.env`，設定 `TRAINING_SMB_SERVER`、`TRAINING_SMB_SHARE` 等（見 `deploy/.env.example`）。
2. `docker compose up -d --force-recreate training-backend`
3. 容器內驗證：`docker compose exec training-backend python -c "from app.config import get_settings; print(get_settings().smb_configured)"`

#### 症狀 B：`0xc0000234`（教材 NAS 登入）

**根因**：AD 帳號鎖定（多次密碼錯誤），非程式設定問題。

**修復**：請 IT 解鎖帳號；解鎖前勿再試登入。

#### 症狀 C：`encrypt_env_secret.py` 報「尚未設定 CREDENTIAL_SECRET」

**根因**：金鑰只寫在 `backend/.env`，或 `TRAINING_CREDENTIAL_SECRET` 未寫入 `deploy/.env` 且容器未 recreate。

**修復**：

```bash
cd /opt/apps/enterprise-portal/deploy
# 一次性加密（金鑰尚未進容器時）
docker compose exec -e CREDENTIAL_SECRET='你的Fernet金鑰' training-backend \
  python scripts/encrypt_env_secret.py "你的SMTP明文"
# 將 TRAINING_CREDENTIAL_SECRET 與 enc: 輸出寫入 deploy/.env 後 force-recreate
```

詳見 [MIGRATION_GUIDE.md](資料庫遷移/MIGRATION_GUIDE.md)「SMTP 密碼密件化」與 [NAS與路徑跨平台慣例.md](NAS與路徑跨平台慣例.md)。

#### 症狀 D：主機 `backend/` 無 `.venv`

**根因**：Docker 主機不建 venv；Python 在容器內。

**修復**：一律使用 `docker compose exec training-backend python ...`，勿使用 `.venv/bin/python3`。

#### 症狀 E：正式環境遷移報 `uv: command not found`

**根因**：誤將 [§5.3](#53-後端虛擬環境損壞無法啟動) 的 `uv venv` 流程用於 Docker 主機。`uv` 僅供本機開發建 venv。

**修復**：勿在主機安裝 `uv` 或建 `.venv`。從 `deploy/` 執行：

```bash
cd /opt/apps/enterprise-portal/deploy
docker compose exec training-backend python migrations/<腳本名>.py
```

#### 症狀 F：考試中心／成績中心空白（正式環境）

**根因**：未執行 `add_exam_retake_authorization.py`；或備份／遷移路徑誤用 `backend/data/` 而非 `${DATA_ROOT}/training/`。

**修復**：見 [生產部署指南](生產部署指南.md) §5.2、[MIGRATION_GUIDE.md](資料庫遷移/MIGRATION_GUIDE.md)「已通過授權重考」。

#### 症狀 G：已考過仍要求報到（正式環境）

**根因**：`attendance_records` 缺列；未執行 `backfill_attendance_from_exam.py`。

**修復**：個人點「立即報到」；維運執行 `docker compose exec training-backend python migrations/backfill_attendance_from_exam.py`。
