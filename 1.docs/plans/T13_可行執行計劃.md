# T13 可行執行計劃（Execution Plan）

## 執行策略

- 以「先決策、再落地、最後擴充」三段式執行，降低一次性改動風險。
- 先交付高可行功能（批次刪除、報到卡片切換、立即報到），再收斂權限與列印規則。
- 每階段都包含可回歸驗證與文件同步，避免需求漂移。

## Phase A：決策凍結（0.5 天）

- 鎖定角色映射與資料範圍規則（主管/稽核=部門內；其餘指定角色=全域）。
- 鎖定列印資料最小集與兩個列印選項行為（簽名、歷程）。
- 鎖定請假判定規則（`absence_reason_code` 有值即請假）。
- 產出決策記錄，作為後續 API 與前端驗收依據。

## Phase B：低風險核心交付（1.5~2 天）

- 報到總覽：固定 active 計畫、統計卡片（應到/實到/未到/請假）、單一清單切換。
- 考試中心：未報到開始考試時顯示提示 Modal，支援「立即報到」後導向考試。

## 風險控制與可行性結論

- 最高風險是角色名稱與資料範圍映射不一致；已在 Phase A 前置消除。
- 第二風險是列印歷程資料完整性；先以現有可得資料交付，歷史補全另開增量需求。
- 依目前程式結構（既有 `report.py`、`training.py`、對應前端頁面）屬可行增修，不需大重構。
