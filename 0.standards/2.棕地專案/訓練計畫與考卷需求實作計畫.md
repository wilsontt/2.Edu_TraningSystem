## 1. 目的與範圍

本計畫涵蓋報到統計 Modal 調整、編輯計畫受課單位連動、考試中心版面、列表隔列 highlight、考卷工坊規格與上傳預覽流程、以及年度訓練合併報到與未到原因填寫等需求。

---

## 2. 訓練計畫管理

### 2.1 報到統計 Modal（TrainingPlanManager.tsx）

- **統計卡片（應到/實到/未到/請假）與單一清單切換**  
  - 報到統計 Modal 上方顯示四張統計卡片：應到、實到、未到、請假。  
  - 點選卡片後，下方只保留一個表格清單，內容依所選卡片切換（預設顯示應到清單）。  
  - 「未到人數」顯示未報到且尚未填寫未到原因者數量；「請假人數」顯示未報到且已填寫未到原因者數量（以 `absence_reason_code` 判斷）。

- **報到 QRcode 與應到人數設定改為左右排列**  
  - 將「報到 QRcode」與「應到人數設定」兩個區塊由上下排列改為同一列左右排列（例如 `grid grid-cols-2` 或 flex），並在 RWD 時可改為上下堆疊。

- **產生 QRcode**  
  - 維持現狀，無需修改。

- **所有列表隔列 highlight**（已實作）  
  - 已於全系統實作：表格/列表之**偶數列**使用 `even:bg-gray-100`（或依 index 奇偶套用 `bg-gray-100`），與既有 hover 並存。涵蓋：訓練計畫管理主表、報到統計 Modal 內單一清單表格（依卡片切換內容）、系統管理（單位/分類/人員/角色/權限/功能清單/QRcode 管理）、成績中心部門績效表與展開後之成員成績/考生成績列表、個人成績歷史、考卷工坊計畫列表與題目列表、題庫維護、個人成績頁之使用者下拉選單。

### 2.2 編輯訓練計畫：受課單位勾選 → 個人受訓對象自動勾選

- **資料來源**  
  - 目前 TrainingPlanManager 的 `users` 來自 `GET /admin/users`，只映射了 `emp_id, name, dept_name`。  
  - 需在呼叫時一併取得使用者的 `dept_id`（若 API 未回傳，需在後端 admin 的 UserDetail/序列化中加入 `department.id`），前端型別改為含 `dept_id`。

- **邏輯**  
  - 當使用者在「受課單位」勾選/取消某部門時：  
    - 勾選部門：將「個人受訓對象」中 `user.dept_id` 屬於該部門的使用者自動勾選（加入 `formData.target_user_ids`）。  
    - 取消部門：將該部門使用者的 emp_id 從 `target_user_ids` 中移除。  
  - 僅在「受課單位」變更時觸發上述同步；使用者手動勾選/取消「個人受訓對象」時不反寫受課單位。

## 4. 考卷工坊

### 4.1 規格與題目格式（1.spec.md）

- **2.1 TXT 匯入格式**  
  - 確認並補齊：**HINT**（提示內容）、**LEVEL**（題目難易度）皆在條列欄位中寫清楚，且與內文範例一致（含半形/全形冒號說明）。  
  - 題目格式範例 (TXT) 中，在既有 Q/A/B/C/D/ANS/SCORE/HINT 之後，明確加上 **LEVEL** 範例（例如 `LEVEL: E` 或 `LEVEL: Medium`，與規格 2.1 一致）。

### 4.2 後端：LEVEL 支援（選填）

- **Parser**：parser.py 新增解析 `LEVEL:` 或 `Level:`（E/M/H 或 Easy/Medium/Hard），寫入 `current_question["level"]`。  
- **資料庫**：若尚未有題目難易度欄位，需在 `questions` 與 `question_bank` 表新增 `level`（可選，VARCHAR 或 ENUM）。  
- **API**：上傳/匯入時傳遞並儲存 `level`；題目列表/詳情回傳時包含 `level`。

### 4.3 上傳 TXT 改為「先預覽、再匯入」

- **流程**  
  1. 使用者選擇檔案後，**不要**直接呼叫現有 `POST /admin/exams/upload`。  
  2. 改為呼叫新 API：例如 `POST /admin/exams/upload/preview`（或 `parse-only`），只解析 TXT、回傳題目列表（content, options, answer, score, hint, level 等），不寫入 DB。  
  3. 前端顯示預覽列表，每題可勾選；提供「加入該訓練計畫題目」「匯入題庫」等按鈕（可多選題目、分開或一併執行）。  
  4. 使用者確認後，再呼叫既有或新寫的 API，將「已勾選題目」寫入該訓練計畫和/或題庫。

- **後端**  
  - 新增 preview 用 endpoint，使用現有 TXTParser 解析，回傳結構化題目陣列。  
  - 可選：新增「僅匯入選定題目至計畫」與「僅匯入選定題目至題庫」的 API，或沿用現有 upload 但改為接受「已解析題目 + plan_id」的 payload，避免重複解析。

- **前端**  
  - ExamStudio.tsx：選擇檔案後改為呼叫 preview API → 開啟預覽 Modal（表格/清單列出題目，每題 checkbox）。預覽中可「全選/取消」「加入該訓練計畫」「匯入題庫」，再送出對應 API。

## 5. 年度訓練：合併報到與未到原因

### 5.1 同時顯示多訓練報到狀況

- **入口**  
  - 在導覽列新增一項「報到總覽」（路徑 `/attendance-overview`），或放在既有「訓練計畫」下、以分頁/區塊呈現「多計畫報到總覽」。  
  - 需有權限控制（與現有報到統計一致，例如 `menu:plan` 或 Admin）。

- **資料與畫面**  
  - 新頁面或新區塊：可依「年度」「訓練大類」等篩選多個訓練計畫，一次列出多個計畫的報到統計（應到/實到/未到/出席率）。  
  - 後端：可新增 `GET /training/attendance/summary?year=2025&category_id=...` 回傳多計畫的報到彙總；或前端多次呼叫既有 `GET /training/plans/{plan_id}/attendance/stats` 組合成總覽。

### 5.2 未報到原因（病假、出差、公假…）

- **資料模型**  
  - 未報到者目前僅在「報到統計」中列出，無儲存原因。  
  - 新增：**未到原因記錄**。建議新增表，例如 `attendance_absence_reasons`：  
    - `id`, `plan_id`, `emp_id`, `reason_code`（對應系統原因選項）, `reason_text`（其他時填寫）, `recorded_by`, `recorded_at`。  
  - **原因選項**：另建設定表或 enum，例如 `absence_reason_types`（id, code, name, is_system），預設：病假、出差、公假、其他（其他時必填 reason_text）。需有後台或設定頁可維護選項。

- **權限**  
  - 僅「未報到」名單可填寫；填寫人為該員主管或有權限者（如 Admin、或具 `menu:plan` 且為該部門主管）。權限邏輯需在 API 與前端一致。

- **API**  
  - 取得某計畫未報到名單時，一併回傳已填寫的未到原因（若已有）。  
  - 新增/更新未到原因：例如 `PUT /training/plans/{plan_id}/attendance/absence-reason`，body：`{ "emp_id", "reason_code", "reason_text?" }`。

- **UI**  
  - 在「報到統計」Modal 的未報到用戶列表中，每列增加「填寫原因」按鈕或下拉選單；點選後可選原因（病假/出差/公假/其他），「其他」時必填文字。  
  - 在「年度報到總覽」中，未報到名單同樣可填寫或顯示已填寫原因。

## 8. 注意事項

- **產生 QRcode**：維持現狀，無需修改。  
- **LEVEL**：若題庫/題目表尚未有 `level` 欄位，需做 DB migration。  
- **未到原因選項**：需確認「有地方設定」是系統參數表、後台 CRUD，還是僅預設 enum；實作時依此決定是否新增設定頁與 API。
